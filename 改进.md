# TicTacToe 改进计划

## 当前状态

C++ 端 ~320 行单文件 (`game/main.cpp`)，职责：
- 跨平台 Socket 封装（Winsock / BSD）
- 终端棋盘渲染
- 输入解析与合法性校验
- 胜负平判定
- CLI 参数解析与游戏主循环

Python 端负责 AI 推理与训练。C++ 作为纯客户端，不涉及任何算法决策。

---

## 改进一：纯 C++ MCTS 博弈引擎

### 目标

在 C++ 端实现蒙特卡洛树搜索（Monte Carlo Tree Search），做到零外部依赖的本地 AI 对局能力。

### 技术点

**MCTS 四阶段：**
- **Selection** — 从根节点出发，按 UCB1 公式递归选择子节点，直到叶节点
- **Expansion** — 在叶节点上扩展一个未尝试过的合法走法
- **Simulation (Rollout)** — 从新节点开始随机走子直至终局，返回胜负结果
- **Backpropagation** — 将结果沿路径回传，更新各节点的 visit 计数和 value 累加

**C++ 实现要点：**
- 节点池（object pool / arena allocator）管理 MCTS 树节点，避免每步 new/delete 开销。棋盘格子 ≤9，每局 MCTS 迭代数可达 10^4~10^5，内存分配是瓶颈
- `std::vector<Node>` 预分配 + 索引替代指针（cache-friendly + 避免 dangling）
- UCB1 公式：`value/visits + C * sqrt(ln(parent_visits) / visits)`
- 模板化棋盘编码：`template<int N>` 支持扩展至 N×N 棋盘
- 可配置模拟次数与探索常数 C，提供性能基准对比

### 新增文件

```
game/
├── ai/
│   ├── mcts.h          # MCTS 引擎头文件
│   ├── mcts.cpp        # 核心实现
│   ├── node_pool.h     # 节点池 / arena allocator
│   └── mcts_bench.cpp  # 性能基准测试
├── core/
│   ├── board.h         # 棋盘状态（从 main.cpp 提取）
│   ├── board.cpp
│   ├── rules.h         # 胜负判定（从 main.cpp 提取）
│   └── rules.cpp
└── main.cpp            # 精简后的入口
```

### 面试可聊

- "为什么用 node pool 而不是 `std::make_unique` 逐个分配？" → cache locality、分配延迟、内存碎片
- "UCB1 的 exploration constant C 怎么选？" → 理论值 √2 与实测调参，exploration vs exploitation 权衡
- "如何验证 MCTS 的正确性？" → 与穷举 minimax 在小棋盘上对比、与 Python 神经网络对弈胜率统计
- "时间复杂度？" → O(iterations × (selection_depth + rollout_depth))，每步棋多少万次模拟
- "如何并行化？" → root parallelization / leaf parallelization / tree parallelization 的取舍

## 改进二：C++ 并发游戏服务器

### 目标

用 C++ 重写服务端，支持多个客户端同时连接并对弈。Python 神经网络作为可选的 AI 后端接入。

### 技术点

**网络层：**
- `epoll`（Linux）/ `IOCP` 或 `select`（跨平台 fallback）实现 I/O 多路复用
- 非阻塞 Socket + 就绪事件循环
- 自定义二进制协议：`[4字节长度][2字节类型][可变payload]`，替代当前文本协议
- 正确处理 TCP 粘包/半包：环形缓冲区 + 状态机解析

**并发模型：**
- 线程池：一个 I/O 线程 + N 个 worker 线程
- `std::thread` + `std::mutex` + `std::condition_variable` 实现生产者-消费者队列
- 每局游戏作为一个 session，worker 线程从队列取 session 处理
- 连接状态机：`LOBBY → MATCHED → IN_GAME → FINISHED`

**对局管理：**
- 玩家匹配队列（FIFO）
- Session 生命周期管理
- 断线检测与超时处理

### 新增文件

```
server/
├── net/
│   ├── socket.h         # Socket RAII 封装
│   ├── event_loop.h     # I/O 多路复用事件循环
│   ├── event_loop.cpp
│   ├── protocol.h       # 二进制协议编解码
│   └── protocol.cpp
├── game/
│   ├── session.h        # 对局会话管理
│   ├── session.cpp
│   ├── matchmaker.h     # 玩家匹配
│   └── matchmaker.cpp
├── thread/
│   └── thread_pool.h    # 线程池
└── main.cpp             # 服务入口
```

### 面试可聊

- "为什么用 epoll 而不是 select？" → FD 数量限制、O(n) vs O(1) 事件扫描
- "TCP 粘包怎么处理？" → 环形缓冲区 + 先读定长头再读变长体的状态机
- "线程池怎么设计？" → 任务队列 + condition_variable，讨论负载均衡与惊群问题
- "二进制协议 vs 文本协议？" → 编解码效率、可读性、扩展性（protobuf/flatbuffers 的考量）
- "如何测试并发服务端？" → 模拟多客户端并发连接、断线重连、乱序消息

## 改进三：构建系统与工程化

### 目标

从单 Makefile 升级到现代 CMake，加入单元测试与基准测试。

### 技术点

- CMake 3.15+：`target_*` 系列命令，PUBLIC/PRIVATE/INTERFACE 依赖管理
- 项目结构：`game/` 编译为静态库 `libtictactoe_game.a`，AI 模块编译为 `libtictactoe_ai.a`，main 和 server 各自链接
- Google Test 集成（FetchContent 自动下载）
- 测试覆盖：棋盘初始化、胜负判定、合法走法、MCTS UCB1 选择、协议编解码
- Google Benchmark 对 MCTS 模拟吞吐量做微基准测试
- clang-format 统一代码风格

### 面试可聊

- "INTERFACE vs PUBLIC vs PRIVATE 的区别？" → 头文件依赖传播
- "为什么用静态库而不是头文件 only？" → 编译时间、符号隔离
- "写了哪些测试？怎么保证 MCTS 正确？" → 与确定性算法交叉验证

## 改进四：Minimax + Alpha-Beta 剪枝（与 MCTS 对照）

### 目标

实现经典博弈树搜索作为 MCTS 的对照实现，用于正确性验证与性能对比。

### 技术点

- Alpha-Beta 剪枝：最优情况下搜索节点数从 O(b^d) 降为 O(b^(d/2))
- 迭代加深（Iterative Deepening）：逐步加深搜索深度，超时则返回当前最优
- 置换表（Transposition Table）：`std::unordered_map<board_hash, TTEntry>` 缓存已搜索局面
- Zobrist Hashing：增量计算棋盘哈希，用于置换表索引
- 走法排序（Move Ordering）：优先搜索历史最佳走法，提升剪枝效率

### 面试可聊

- "Alpha-Beta 剪枝的最坏/最优复杂度？" → 走法排序对剪枝效率的影响
- "Zobrist Hashing 怎么处理冲突？" → 64-bit key 冲突概率、不存完整棋盘只在冲突时回退重算
- "MCTS vs Minimax 各适合什么场景？" → 分支因子大/小、评估函数易/难定义、实时性要求
- "为什么井字棋还需要置换表？" → 虽然棋盘小（3^9 ≈ 19683），但在更大棋盘扩展中体现价值

## 改进五：SDL2 / Raylib 图形化前端

### 目标

在保留终端版本的同时，增加一个图形化客户端，展示同一套游戏逻辑库的复用能力。

### 技术点

- SDL2 或 raylib：跨平台窗口创建 + 2D 渲染
- 棋盘绘制：网格 + 棋子图形 + 高亮最近一步
- 鼠标点击落子 + 键盘快捷键
- 同一套 `libtictactoe_game.a` 链接到 GUI 和 CLI 两个前端
- 实时显示 MCTS 模拟进度与胜率评估

### 面试可聊

- "同一个库怎么给终端和 GUI 共用？" → 设计时关注点分离，核心逻辑不依赖 I/O
- "渲染循环 vs 事件循环？" → 游戏循环的帧率控制与输入处理

## 优先级与实施顺序

| 优先级 | 改进项 | 理由 |
|--------|--------|------|
| P0 | 改进一：MCTS 博弈引擎 | C++ 端新增核心算法能力，彻底改变"C++ 只做渲染"的现状 |
| P0 | 改进三：CMake + 单元测试 | 后续所有改进的基础，先搭架子再填内容 |
| P1 | 改进四：Minimax + Alpha-Beta | 与 MCTS 形成对照，正确性验证，且是经典面试题素材 |
| P2 | 改进二：并发游戏服务器 | 网络编程 + 多线程深度，但工作量较大 |
| P3 | 改进五：GUI 前端 | 展示库复用，但技术增量主要在 SDL/raylib 使用而非 C++ 核心能力 |

### 推荐起步路径

```
1. CMake 重构 → 提取 core/board rules → libtictactoe_core.a
2. 基于 core 库写 Google Test 单元测试
3. 在 ai/ 下实现 MCTS → libtictactoe_ai.a
4. MCTS vs MCTS 自弈验证 + Benchmark
5. Minimax + Alpha-Beta 对照实现
6. （可选）C++ server / GUI
```

完成 P0 后，C++ 端从 ~320 行单文件终端渲染变为：
- `core/` — 可复用的游戏逻辑库（~200 行）
- `ai/mcts` — MCTS 引擎（~400 行）
- `ai/minimax` — Minimax 搜索（~250 行）
- `tests/` — 单元测试（~300 行）
- `CMakeLists.txt` — 现代构建系统

总计 ~1500 行 C++，涵盖算法、数据结构、内存管理、测试。
